## 오늘 나눈 이야기 요약

로컬 LLM 과 더 똑똑하게 사용하기 위해 **LLM Wiki** 개념에 대해 브레인스토밍했습니다.

## 핵심 아이디어

### RAG 의 한계

현재 대부분의 RAG(Retrieval-Augmented Generation) 시스템은 다음과 같은 문제가 있습니다:

- 매번 원문에서 관련 청크를 찾아서 LLM 에 주지만, **지식이 누적되지 않음**
- 같은 질문을 반복할 때마다 동일한 문서 스캔
- 지식의 구조화와 통합이 없음

### LLM Wiki 해결책

LLM 이 지식을 **구조화된 Markdown 위키**로 "컴파일"해서 **누적**합니다:

- 새로운 소스를 추가하면 LLM 이 핵심 정보를 추출하고 기존 위키에 통합
- 엔티티 페이지 업데이트, 주제 요약 수정, 모순되는 주장 식별
- 지식이 계속 쌓이고 개선됨

## 3층 아키텍처

```
Raw Sources → Wiki → Schema
```

1. **Raw Sources** - 원본 소스 (문서, 대화, 링크)
2. **Wiki** - 컴파일된 지식베이스 (Markdown 파일)
3. **Schema** - 지식 구조화 (태그, 링크, 관계)

## 3가지 연산

### 1. Ingest (컴파일)
- 새로운 소스를 추가하면 LLM 이 핵심 정보 추출
- 기존 위키에 통합 (업데이트, 수정, 연결)

### 2. Query (검색)
- 자연어 질문 → 위키에서 관련 정보 검색
- RAG 보다 더 구조화된 답변 가능

### 3. Lint (정비)
- 낡은 정보 식별 및 업데이트
- 중복 제거, 링크 점검, 구조 최적화

## Andrej Karpathy 의 사례

Andrej Karpathy (OpenAI 공동창업자, 전 Tesla AI 디렉터, 현재 Anthropic 프리트레이닝 팀) 가 2026 년 4 월 GitHub Gist 에 공개한 LLM Wiki 개념이 큰 반향을 일으켰습니다.

### Karpathy 의 핵심 통찰

> "LLM 이 원본 문서를 매번 다시 읽는 대신, 한 번 컴파일된 구조화된 위키를 만들고 계속 업데이트하라."

### Karpathy 의 비유: "컴파일"

소프트웨어 엔지니어링의 컴파일 개념을 차용했습니다:

- **Raw 코드** → **컴파일된 바이너리** (RAG 의 원문 → 청크 검색)
- **컴파일된 위키** → **실행 가능한 지식** (LLM Wiki)

### Karpathy 의 구현

- GitHub Gist 에 전체 아키텍처 공개: https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
- `raw/` 폴더의 모든 데이터를 요약하고 백링크 생성
- 개념별로 카테고리화하고 기사 작성, 링크 연결
- Git 을 통해 버전 관리

### Karpathy 의 LLM Wiki 특징

1. **엔티티 페이지** - 사람, 개념, 프로젝트별로 페이지 분리
2. **주제 요약** - 각 주제에 대한 최신 요약 유지
3. **모순 처리** - 새로운 정보가 기존 주장과 충돌하면 업데이트
4. **프로버넌스 추적** - 정보 출처 추적 가능
5. **Git 버전 관리** - 모든 변경 사항 기록

## 왜 이런 구현 방향을 선택했는가?

Karpathy 의 아이디어를 실제로 구현하면서 다음과 같은 기술 스택을 선택하게 된 이유를 기록합니다.

### 1. Hugo + PaperMod (정적 블로그)

**선택 이유:**
- **빠른 로딩 속도**: 정적 사이트는 동적 사이트보다 훨씬 빠름
- **무료 호스팅**: GitHub Pages 와 완벽하게 통합
- **깔끔한 디자인**: PaperMod 테마의 미니멀한 UI
- **Git 연동**: Markdown 파일로 콘텐츠 관리 → Git 버전 관리와 자연스럽게 통합
- **보안**: 정적 파일이므로 서버 해킹 위험 없음

**대안 비교:**
- WordPress: 설정 복잡, 보안 위험, 유료 호스팅 필요
- Notion: 클라우드 종속, 오프라인 불가, API 제한
- Obsidian: 로컬 전용, 호스팅 별도 필요

### 2. Wiki 구조 (오류 해결, 기술 결정, 도구 사용법)

**선택 이유:**
- **RAG 의 한계 극복**: 매번 원문 스캔 대신 구조화된 지식 축적
- **주제별 분류**: 오류 해결, 기술 결정, 도구 사용법 등 카테고리화
- **누적 효과**: 지식이 계속 쌓이고 개선됨
- **검색 용이**: 태그와 링크로 빠른 검색 가능

**Karpathy 의 3 층 아키텍처 적용:**
- `content/wiki/errors/` → Raw Sources (원본 오류 기록)
- `content/wiki/decisions/` → Wiki (컴파일된 결정 사항)
- `content/wiki/tools/` → Schema (도구 구조화)

### 3. GitHub Pages + GitHub Actions

**선택 이유:**
- **무료 호스팅**: 개인 프로젝트에 최적
- **CI/CD 자동화**: 코드 푸시 시 자동 빌드 및 배포
- **Git 연동**: 모든 변경 사항 추적
- **커뮤니티**: Hugo + GitHub Pages 는 가장 인기 있는 정적 블로그 조합

**자동화 흐름:**
```
코드 푸시 → GitHub Actions 빌드 → public 폴더 생성 → GitHub Pages 배포
```

### 4. pi 에이전트 연동

**선택 이유:**
- **Markdown 자동 생성**: pi 가 Markdown 파일을 자동으로 생성
- **개발 효율화**: 수동 파일 생성 대신 AI 가 콘텐츠 작성
- **Git push 연동**: 생성된 파일 → Git 커밋 → 자동 배포

**연동 구조:**
```
사용자 요청 → pi 가 Markdown 파일 생성 → Git 커밋 → GitHub Actions 빌드 → GitHub Pages 배포
```
